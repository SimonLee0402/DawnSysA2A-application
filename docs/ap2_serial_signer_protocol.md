# AP2 Serial Signer Protocol

## Purpose

This document defines the desktop-to-MCU serial protocol for Dawn's AP2 signer path.

Current status on March 11, 2026:

- `dawn-node ap2 signer serial ...` is already available as the desktop-facing product surface.
- Real MCU transport is not implemented yet because hardware has not been purchased or flashed.
- The desktop CLI currently supports a `serial` development mode backed by `DAWN_AP2_SERIAL_MOCK_SEED_HEX` so the user journey, payload shape, and approval flow can be validated before hardware arrives.

The goal of this document is to freeze the wire contract now, so firmware can implement against it later without changing the CLI command model.

## Scope

This protocol covers:

- serial handshake between the desktop CLI and an AP2 signing device
- public-DID discovery
- AP2 payload signing
- health checks and error reporting

This protocol does not cover:

- firmware flashing
- USB VID/PID provisioning
- secure element selection
- bootloader/update transport

## Product Mapping

The protocol backs these existing CLI commands:

- `dawn-node ap2 signer serial COM7 --baud 115200 --protocol dawn-ap2-v1`
- `dawn-node ap2 signer status`
- `dawn-node ap2 prepare <transaction_id>`
- `dawn-node ap2 sign <transaction_id>`
- `dawn-node ap2 approve-local <transaction_id>`

Current development fallback:

- `dawn-node ap2 signer serial COM7 --baud 115200 --mock-seed-hex <hex>`

That fallback must keep producing the same payload and result fields that the real serial device will produce:

- `payload`
- `mcuPublicDid`
- `mcuSignature`

## Transport

- Medium: UART-style serial transport exposed as a local COM port
- Default baud rate: `115200`
- Data bits: `8`
- Parity: `none`
- Stop bits: `1`
- Flow control: `none`
- Character encoding: UTF-8
- Application framing: one JSON object per line delimited by `\n`

Rationale:

- line-delimited JSON is easy to inspect with terminal tools during bring-up
- it works well for the current CLI and mock signer path
- it avoids a second binary framing layer before hardware exists

Each request and response must fit on a single line. Devices should reject oversized payloads rather than streaming partial frames.

## Protocol Version

- Protocol identifier: `dawn-ap2-v1`
- CLI config field: `DAWN_AP2_SERIAL_PROTOCOL`
- Every device response must echo the protocol version in the `protocol` field

If the device speaks a different protocol version, the CLI must fail closed.

## Message Envelope

Every request from the desktop CLI to the signer uses this envelope:

```json
{
  "protocol": "dawn-ap2-v1",
  "requestId": "6c5f6f87-1104-4f91-8f2a-4ac0a77f4a1d",
  "command": "ping",
  "params": {}
}
```

Every response from the signer uses this envelope:

```json
{
  "protocol": "dawn-ap2-v1",
  "requestId": "6c5f6f87-1104-4f91-8f2a-4ac0a77f4a1d",
  "ok": true,
  "result": {},
  "error": null
}
```

Rules:

- `requestId` is generated by the desktop CLI and must be echoed unchanged
- exactly one of `result` or `error` must be populated
- unknown top-level fields must be ignored
- unknown commands must return `ok=false`

## Commands

### `hello`

Used immediately after opening the port.

Request:

```json
{
  "protocol": "dawn-ap2-v1",
  "requestId": "9f17d2ef-2c22-4d41-a6c8-2789af98a6fc",
  "command": "hello",
  "params": {
    "client": "dawn-node",
    "clientVersion": "desktop-cli"
  }
}
```

Success response:

```json
{
  "protocol": "dawn-ap2-v1",
  "requestId": "9f17d2ef-2c22-4d41-a6c8-2789af98a6fc",
  "ok": true,
  "result": {
    "device": "dawn-ap2-signer",
    "firmwareVersion": "0.1.0",
    "supports": ["ping", "get_public_did", "sign_payload"]
  },
  "error": null
}
```

### `ping`

Cheap liveness check.

Request:

```json
{
  "protocol": "dawn-ap2-v1",
  "requestId": "e9d6d65a-8ecb-4d1f-87c5-638b8d49e6de",
  "command": "ping",
  "params": {}
}
```

Success response:

```json
{
  "protocol": "dawn-ap2-v1",
  "requestId": "e9d6d65a-8ecb-4d1f-87c5-638b8d49e6de",
  "ok": true,
  "result": {
    "timestampUnixMs": 1773259200000
  },
  "error": null
}
```

### `get_public_did`

Returns the public AP2 identity without signing a payload.

Request:

```json
{
  "protocol": "dawn-ap2-v1",
  "requestId": "28a0f7ca-a8d0-4300-8c20-d2d3690a3b7c",
  "command": "get_public_did",
  "params": {}
}
```

Success response:

```json
{
  "protocol": "dawn-ap2-v1",
  "requestId": "28a0f7ca-a8d0-4300-8c20-d2d3690a3b7c",
  "ok": true,
  "result": {
    "mcuPublicDid": "did:dawn:mcu:ab12cd34..."
  },
  "error": null
}
```

### `sign_payload`

Signs the exact AP2 payload string that Dawn already prepares in the CLI and gateway-facing approval flow.

Current payload format:

```text
{transaction_id}:{mandate_id}:{amount_with_4_decimals}:{description}
```

Example:

```text
0f6d58f2-35aa-41d8-bc57-3df01b7bc0fe:9d2d3728-716c-4c07-84df-1c3d26dcbef2:12.5000:hotel booking
```

Request:

```json
{
  "protocol": "dawn-ap2-v1",
  "requestId": "b8ecb7cc-8c56-4df5-8b4b-52bd87e7d728",
  "command": "sign_payload",
  "params": {
    "payload": "0f6d58f2-35aa-41d8-bc57-3df01b7bc0fe:9d2d3728-716c-4c07-84df-1c3d26dcbef2:12.5000:hotel booking",
    "encoding": "utf-8",
    "algorithm": "ed25519"
  }
}
```

Success response:

```json
{
  "protocol": "dawn-ap2-v1",
  "requestId": "b8ecb7cc-8c56-4df5-8b4b-52bd87e7d728",
  "ok": true,
  "result": {
    "mcuPublicDid": "did:dawn:mcu:ab12cd34...",
    "mcuSignature": "5f0d4a...",
    "algorithm": "ed25519"
  },
  "error": null
}
```

Signing rules:

- the signer must sign the exact UTF-8 bytes of `payload`
- the device must not trim, normalize, or rewrite whitespace
- the returned signature must be lowercase hex
- the returned DID must remain stable across calls for the same keypair

## Error Model

On failure, the device returns:

```json
{
  "protocol": "dawn-ap2-v1",
  "requestId": "b8ecb7cc-8c56-4df5-8b4b-52bd87e7d728",
  "ok": false,
  "result": null,
  "error": {
    "code": "device_locked",
    "message": "signer requires operator unlock"
  }
}
```

Defined error codes:

- `unsupported_protocol`
- `unsupported_command`
- `invalid_request`
- `invalid_payload`
- `device_locked`
- `key_unavailable`
- `sign_failed`
- `busy`
- `internal_error`

CLI handling expectations:

- `unsupported_protocol` is fatal and should instruct the user to reconfigure the signer protocol
- `busy` may be retried once after a short delay
- `device_locked`, `key_unavailable`, and `sign_failed` must fail closed and must not auto-approve AP2

## Timeout Model

- Port open timeout: `3s`
- Initial `hello` timeout: `3s`
- `ping` timeout: `2s`
- `get_public_did` timeout: `3s`
- `sign_payload` timeout: `10s`

If a timeout occurs, the CLI should surface it as a transport failure and keep the AP2 transaction pending.

## Session Rules

- The CLI opens the port on demand, performs `hello`, runs the requested command, then closes the port.
- Devices should be stateless across requests where possible.
- If the device needs an unlock step later, that should be represented as an error or an added command in `dawn-ap2-v2`, not by changing the `sign_payload` response shape in `v1`.

## Implementation Notes

Desktop CLI behavior today:

- `local_seed` mode signs with `DAWN_AP2_MCU_SEED_HEX`
- `serial` mode signs with `DAWN_AP2_SERIAL_MOCK_SEED_HEX` when staged
- `serial` mode without a mock seed intentionally fails closed

That means the command surface and approval loop are stable now, while the transport implementation remains intentionally deferred until hardware exists.

## Next Implementation Step

When hardware arrives, the next code change should be limited to:

1. add a real serial transport behind `sign_ap2_payload(...)`
2. execute `hello`
3. execute `sign_payload`
4. map the device response into the existing `SignedAp2Payload`

No CLI command redesign should be needed.
